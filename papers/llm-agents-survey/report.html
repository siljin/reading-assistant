Plain-language summary

This Fudan survey explains why LLMs changed AI agents from brittle task-specific systems into plausible general assistants. Its core map is Brain (reasoning, memory, planning), Perception (text, image, audio adapters), and Action (tool calls, text output, or embodied movement). It then surveys single-agent workflows, multi-agent systems, human-agent teams, and early 'agent societies.' The main verdict: tool-using knowledge agents are close to useful today, while robust evaluation, physical-world grounding, hallucination control, and safety remain unresolved.

problem context

Why AI agents, and why now?

LLMs provide a flexible cognitive core that can be wired into tools, memory, sensors, and action systems.

Older agents depended on brittle specialists for vision, planning, language, and control. Each new task required new glue, and systems that worked in one domain often broke in another. LLMs changed the equation by combining broad knowledge, instruction following, and multi-step reasoning in one model. Once an LLM can decide what tool to call next, it becomes the center of a general agent loop. This survey captures that inflection point.

  • Symbolic, reactive, and RL agents each hit a generalisation ceiling
  • LLMs combine knowledge, instruction-following, and reasoning
  • LLM + tools + memory is the core agent architecture

core contribution

What the survey actually contributes

Its value is a shared Brain-Perception-Action map for a fast-moving field, not a new algorithm.

The survey is taxonomic. The authors read roughly 300 papers through mid-2023 and group them into three layers: Brain (memory, reasoning, planning), Perception (adapters for text, image, and audio), and Action (text, tool calls, or embodied control). That shared vocabulary matters because the field had many demos but little structure. The paper also contributes a curated reading list, but it runs no experiment and cannot settle which cited systems are best.

  • Proposes Brain-Perception-Action as a three-layer framework
  • Organises ~300 papers into a coherent taxonomy
  • Surfaces the missing unified benchmark

The highest-leverage research problem is evaluation, not another demo.

  • Agent safety must be evaluated across model behavior, tool permissions, memory, and human checkpoints.

architecture or pipeline

Inside the three layers: Brain, Perception, Action

Most engineering pressure sits in the Brain layer: memory, planning, reasoning, and tool orchestration.

Brain. The LLM handles interaction, knowledge retrieval, memory, reasoning, planning, and generalisation. The hard parts are context limits and hallucination; retrieval, summaries, and structured memory help but do not solve them. Perception. Text is native. Images and audio usually pass through specialist encoders bridged into the LLM. Action. The agent can write text, call tools, or issue high-level plans to a robot or simulator. Tool use is the mature path; physical embodiment is still research-grade.

  • Memory bottleneck: context window → retrieval or summarisation
  • Vision bridge: specialist encoder → LLM-readable representation
  • Tool use: structured API call → observation → next reasoning step

results interpretation

What the applications actually show

Tool agents are closest to use today; multi-agent systems add power and failure modes; embodied agents remain early.

Single-agent applications work best when tasks are well-defined and tool-accessible: research, coding, web navigation, and domain workflows. Multi-agent applications such as ChatDev and MetaGPT decompose work across roles and often beat single-agent baselines on coding tasks, but communication overhead creates new failure paths. Human-agent cooperation is safer when humans approve consequential actions. Agent societies such as Smallville are the most surprising result: many simple agents can produce emergent social behavior, but this is still speculative.

  • Single-agent systems work best for tool-accessible knowledge tasks
  • Multi-agent coding systems can beat single-agent baselines
  • Smallville shows emergent social behavior, but in a simplified world

limitations and caveats

What the paper gets wrong or doesn't say

The survey names big technical gaps, but underplays cost, production failure rates, and safety evidence.

Stated limitations: no unified benchmark, weak physical grounding, hallucinations, misuse risk, and scaling problems in multi-agent systems. Unstated limitations: production cost is mostly absent, most cited systems are demos, deployment failure rates are unknown, and the taxonomy is the authors' framework rather than a field standard. The paper also reflects mid-2023 model capabilities, so some frontier-model claims have aged quickly. Most importantly, 'the agent completed the task' is not the same as 'the agent completed it safely.'

  • Compute cost is absent
  • Demo ≠ deployment: failure rates and maintenance are not covered
  • Framework is authors' taxonomy, not a community standard
  • Human safety evaluation is sparse

real world implications

What this means for builders and researchers

Build supervised tool agents first. Treat multi-agent and physical autonomy as higher-risk bets.

For builders: ReAct-style tool loops and frameworks like LangChain, AutoGen, and CrewAI make useful agents possible quickly. The hard part is reliability: loops, hallucinated tool calls, bad evidence, and costly actions. Give agents deterministic tools, require evidence, and add human approval for consequential steps. For researchers: evaluation and safety are the infrastructure gaps. For product leaders: domain-specific agents with verified tool access are sellable, but liability is still a product and legal problem.

  • ReAct + tools + guardrails = viable knowledge-work automation
  • Evaluation is the field's biggest unsolved infrastructure problem
  • Liability and accountability are still unresolved
