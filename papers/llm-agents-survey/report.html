Plain-language summary

Before LLMs, building a general AI agent meant stitching together dozens of narrow specialist models, each brittle in its own way. This 86-page survey from Fudan University maps what has changed: LLMs provide a single, flexible cognitive core that can be plugged into perception inputs (text, images, audio) and action outputs (text, tool calls, robot actuators). The authors organise the field around three architectural layers — Brain (the LLM doing reasoning, memory management, and planning), Perception (multimodal input adapters), and Action (tool use, embodied movement). They then cover applications: single-agent workflows like AutoGPT, multi-agent systems where LLMs cooperate or compete, and human-agent teams. A late chapter on 'agent societies' looks at what happens when many agents coexist — emergent social norms, information propagation, even ethical risks. The survey closes with candid open problems: there is no good unified benchmark for agents, physical-world grounding remains hard, and LLM hallucinations and adversarial attacks are unresolved.

problem context

Why AI agents, and why now?

Narrow AI systems could do one thing well. LLMs finally provide a general-purpose cognitive substrate that can be wired into sensors and actuators — making general AI agents a plausible engineering project, not just a sci-fi idea.

For decades, building an AI agent meant assembling brittle specialists: one model for vision, another for planning, another for language. Each new task required re-engineering the glue. The field kept hitting the 'generalisation wall' — systems that crushed benchmarks in one domain collapsed on slightly different inputs. Large language models trained on internet-scale text changed the equation. They store vast common-sense knowledge, follow novel instructions zero-shot, and can reason through multi-step problems when prompted carefully. Researchers quickly realised: if you can get an LLM to reason about what tool to call next, you have the seeds of a general agent. This survey arrives at exactly that inflection point, cataloguing a field that went from a few demos to hundreds of papers in under two years.

  • Classical AI agents (symbolic, reactive, RL-based) each hit a generalisation ceiling
  • LLMs offer knowledge, instruction-following, and reasoning in a single model
  • The combination of LLM + tools + memory = first credible general-purpose agent architecture
  • Field grew from ~0 to hundreds of papers between 2022 and mid-2023

core contribution

What the survey actually contributes

A unified Brain-Perception-Action taxonomy that organises 300+ papers into a coherent field — and a frank account of what is still missing.

The paper's primary contribution is taxonomic, not experimental. The authors read roughly 300 papers published through mid-2023 and propose a three-layer framework: Brain (the LLM core, handling memory, reasoning, and planning), Perception (adapters that convert text/image/audio into LLM-readable form), and Action (the output layer — text generation, tool calls, or embodied robot commands). They then map every major application and open problem onto this framework. This is genuinely useful because the field had no shared vocabulary — every paper invented its own architecture diagram. The secondary contribution is a curated reading list (GitHub repo) that has become a go-to entry point. What the survey does not do: it proposes no new technique, runs no experiment, and cannot adjudicate between competing claims in cited papers.

  • Proposes Brain-Perception-Action as the canonical three-layer framework
  • Organises ~300 papers into a coherent taxonomy for the first time
  • Surfaces the evaluation gap: no unified agent benchmark exists
  • Curated GitHub reading list widely used as field entry point

so what

What this means beyond the paper

The same paper should mean different things to researchers, product builders, and business decision-makers.

The highest-leverage research problem is evaluation, not another agent demo.

Implications
  • The field needs shared benchmarks for utility, sociability, safety, and continual evolution.
  • Agent safety should be evaluated as a system property: model behavior, tool permissions, memory, and human checkpoints together.
  • Multi-agent communication protocols are still informal; standard protocols could become a research infrastructure layer.
Open Questions
  • How do we measure whether an autonomous agent is genuinely better than a scripted workflow?
  • How should benchmarks capture long-horizon failures such as loops, hidden cost, and unsafe tool calls?
  • Can memory systems support continual learning without accumulating stale or harmful beliefs?
Next Actions
  • Design benchmark suites that include cost, failure recovery, and safety outcomes.
  • Compare single-agent and multi-agent systems on the same non-coding tasks.
  • Study guardrail mechanisms as first-class agent architecture components.

Build supervised tool agents now; avoid pretending full autonomy is ready.

Opportunities
  • Research assistants, coding support, document processing, data analysis, and internal operations agents.
  • Domain-specific agents that use verified tools and cite evidence before taking action.
  • Human-in-the-loop workflows where the agent prepares work and the user approves consequential steps.
Guardrails
  • Require evidence before tool calls that affect money, users, legal commitments, or production systems.
  • Constrain tools by permission, audit every action, and provide rollback paths.
  • Use retrieval and deterministic APIs to reduce hallucination, but do not treat them as complete safety solutions.
Next Actions
  • Pilot one narrow internal workflow with clear success metrics and human review.
  • Track agent loop failures, tool-call errors, cost per completed task, and user trust.
  • Delay physical-world autonomy and high-liability decisions until evaluation catches up.

The business wedge is reliability-as-a-service, not generic autonomous agents.

Market Openings
  • Agent-as-a-Service for regulated or domain-specific workflows where tool access and auditability matter.
  • Evaluation and observability platforms for agent behavior, cost, and safety.
  • Vertical products that package an agent with workflow integration, permissions, and accountability.
Adoption Blockers
  • Unclear liability when an agent books, sends, buys, deploys, or advises incorrectly.
  • High operating cost for long-running GPT-4-class agent loops.
  • Lack of trusted benchmarks makes vendor claims hard to compare.
Risks
  • A demo-first product can fail in production because agent errors compound over long tasks.
  • Safety failures can create reputational and legal exposure faster than a normal software bug.
  • Generic agent platforms may be hard to defend if domain workflow data and integrations are weak.
Next Actions
  • Position products around accountable automation, not unrestricted autonomy.
  • Make audit logs, approvals, and permission boundaries visible product features.
  • Use the survey as a market map: tool agents first, multi-agent workflows second, embodied autonomy later.
architecture or pipeline

Inside the three layers: Brain, Perception, Action

The brain layer is where most of the interesting engineering happens — memory management, chain-of-thought planning, and tool orchestration. Perception and action are mostly solved by plugging in existing specialist models.

Brain. The LLM handles five functions: natural language interaction, knowledge retrieval (from training or external DBs), memory (short-term context window + long-term vector stores), reasoning (chain-of-thought, self-consistency, plan-and-execute), and generalisation (zero-shot, few-shot, in-context learning). The hard problems are (1) context window limits — solved by summarisation, vector retrieval, or hybrid memory stores like ChatDB — and (2) hallucination — partially mitigated by tool use and retrieval augmentation, not yet solved. Perception. Textual input is native. Visual input uses pre-trained vision encoders (ViT, CLIP) bridged to the LLM via a projection layer or Q-Former (BLIP-2 style). Audio uses audio spectrogram transformers or cascading specialist models (AudioGPT). The pattern is consistent: freeze a specialist encoder, train a lightweight bridge. Action. Three output modes: text generation (native), tool use (the LLM outputs structured calls to APIs, search engines, code interpreters — ToolFormer-style), and embodied action (the LLM generates high-level plans; a low-level policy or robot API executes them). Tool use is the most mature. Embodied action in the physical world remains research-grade.

  • Memory bottleneck: context window → solved by vector DB retrieval or summarisation
  • Vision bridge: freeze ViT encoder, train small projection layer to LLM space
  • Tool use: LLM outputs structured API call, gets result, reasons over it (ReAct pattern)
  • Embodied: LLM = high-level planner; low-level policy/robot API = executor

results interpretation

What the applications actually show

Single-agent tool use works well enough to automate knowledge work today. Multi-agent pipelines outperform single agents on complex tasks but add failure modes. Embodied agents work in Minecraft; physical-world robots are 5 years behind.

Single-agent applications span three categories. Task-oriented (AutoGPT, web navigation, household task agents): reliable for well-defined, tool-accessible tasks; unreliable when implicit common-sense knowledge is needed. Innovation-oriented (ChemCrow for chemistry, code-generation agents): domain tools are the key enabler; agents can run real experiments via robotic APIs. Lifecycle-oriented (Voyager in Minecraft): lifelong skill accumulation works in closed simulated worlds; real-world transfer is unsolved. Multi-agent applications: cooperative systems (ChatDev, MetaGPT) decompose software projects across specialised roles and consistently beat single-agent baselines on coding benchmarks. Adversarial systems (debate-style) improve factuality and robustness. Human-agent cooperation: instructor-executor paradigm (human gives high-level goal, agent executes) and equal-partnership paradigm (human and agent co-create) both show productivity gains in early studies. Agent societies: Generative Agents (Smallville) showed 25 agents exhibit emergent social behaviours — information spread, relationship formation — without explicit programming. This is the most surprising and speculative result in the survey.

  • AutoGPT and similar single-agent systems work for research/knowledge tasks; fail on embodied common-sense
  • Multi-agent coding (ChatDev, MetaGPT) outperforms GPT-4-solo on software benchmarks
  • Voyager: skill library + iterative prompts → state-of-the-art Minecraft exploration
  • Generative Agents (Smallville): emergent social phenomena from 25 simple agents — most surprising result

limitations and caveats

What the paper gets wrong or doesn't say

The survey is honest about technical gaps but sidesteps compute costs, real deployment failure rates, and the fact that most cited 'results' are demos without rigorous evaluation.

Stated limitations (credit to authors): No unified benchmark, physical-world grounding unsolved, hallucination unresolved, misuse risks real and underspecified, agent scaling introduces communication overhead and emergent risks. Unstated limitations: First, compute cost is invisible. Running GPT-4 as an agent backbone for hours is expensive; the survey never discusses what production-scale agent workloads cost. Second, almost all cited results are demos or academic prototypes; real deployment failure rates (how often agents get stuck, loop, or cause harm) are not reported. Third, the Brain-Perception-Action taxonomy is the authors' own construction — it is elegant but not a field consensus, and some systems (e.g. reward-model-guided agents) do not fit neatly. Fourth, the survey was written in mid-2023; GPT-4 was the frontier model. By late 2024, Claude 3.5 Sonnet, Gemini 1.5, and open-weight models like Llama 3 have significantly shifted the capability landscape. Fifth, human evaluation of agent safety and alignment is sparse in the cited literature — 'the agent completed the task' is not the same as 'the agent completed it safely'.

  • Compute costs for agent workloads: completely absent from the survey
  • Demo ≠ deployment: failure rates, maintenance burden, adversarial resilience in production not covered
  • Framework is authors' taxonomy, not a community standard
  • Field has moved fast: written mid-2023, already dated on model capabilities by 2024
  • Human safety evaluation sparse in cited papers

real world implications

What this means for builders and researchers

Tool-using LLM agents are ready to automate knowledge work today with the right guardrails. Multi-agent pipelines add power but also failure modes. Physical robots and true autonomy are still 5+ years out.

For builders today: The ReAct pattern (Reason + Act loop) and its derivatives (LangChain Agents, AutoGen, CrewAI) give you a production-adjacent agent scaffold in days, not months. The reliability problem — agents that loop, hallucinate tool calls, or confidently do the wrong thing — is the main engineering challenge. Treat the LLM as an unreliable junior engineer: give it deterministic tools (search, code exec, DB queries), require it to show evidence before committing, and build human-in-the-loop checkpoints for high-stakes decisions. For researchers: The evaluation gap is the most impactful open problem. A well-designed agent benchmark suite analogous to MMLU for reasoning or HumanEval for code would unlock the field. Agent safety and alignment (not just capability) is underinvested. For product leaders: Agent-as-a-Service is a real business model — domain-specific agents with verified tool access are sellable now. The risk is reliability and liability: when an agent makes a costly mistake (sends a wrong email, books a wrong flight), who is responsible? This is a legal and product design question, not just a technical one.

  • ReAct + tool use + guardrails = viable knowledge-work automation today
  • Evaluation is the field's biggest unsolved infrastructure problem
  • Physical robotics and full autonomy: 5+ year horizon for production reliability
  • Liability and accountability are the unsolved product/legal questions for deployed agents

Three papers to read next

Start with the ReAct paper to understand the core reasoning-action loop, then Generative Agents for multi-agent dynamics, then ToolFormer for how tool use is taught.

1

ReAct: Synergizing Reasoning and Acting in Language Models

The foundational paper for the Reason + Act loop that almost every LLM agent uses. Understanding ReAct gives you the vocabulary to read every other agent paper.

Read for: Core architecture: how to interleave thought, action, and observation in a single prompt

2

Generative Agents: Interactive Simulacra of Human Behavior

The Smallville paper — 25 agents in a simulated town exhibiting emergent social behaviour. The most striking demonstration of multi-agent dynamics and the closest thing to 'agent society' in the survey.

Read for: Memory architecture, agent personalisation, emergent social phenomena

3

Toolformer: Language Models Can Teach Themselves to Use Tools

Shows how tool use can be learned, not just prompted. Key for understanding the difference between zero-shot tool use (ReAct) and learned tool use (Toolformer).

Read for: Self-supervised tool-use learning, API call generation, training signal design

Question 1

In the Brain-Perception-Action framework, which module is primarily responsible for handling the LLM's inability to remember past conversations beyond its context window?

  1. The Perception module, by compressing multimodal inputs
  2. The Brain module's memory systems — summarisation, vector stores, and external databases
  3. The Action module's tool-use capability
  4. A dedicated memory LLM running in parallel
Show answer

Correct: The Brain module's memory systems — summarisation, vector stores, and external databases

The paper dedicates section 3.1.3 to memory within the Brain module, covering three strategies: summarising historical context into shorter representations, compressing memories into vector embeddings retrieved by similarity, and using external structured databases (ChatDB, DB-GPT) that the LLM queries via SQL. This is a Brain-layer problem, not a Perception or Action problem.

Question 2

Why did symbolic AI agents, which could reason explicitly and were interpretable, eventually give way to LLM-based approaches?

  1. Symbolic agents were too slow to run on 1980s hardware
  2. They required exhaustive hand-coded rules and could not generalise to novel domains or handle uncertainty
  3. The legal liability of rule-based systems was too high
  4. Symbolic agents could not produce natural language output
Show answer

Correct: They required exhaustive hand-coded rules and could not generalise to novel domains or handle uncertainty

Symbolic agents encoded knowledge as logical rules, which meant every new domain needed new rules. They also struggled with uncertainty and large-scale real-world problems because symbolic reasoning is computationally intractable at scale. LLMs sidestep this by learning statistical representations from data, giving them implicit generalisation.

  1. Use a larger LLM to reduce errors
  2. Build a human-in-the-loop checkpoint before any booking is confirmed
  3. Log all agent actions to a database
  4. Use chain-of-thought prompting to make the agent explain itself

Correct: Build a human-in-the-loop checkpoint before any booking is confirmed

The survey highlights the instructor-executor and equal-partnership paradigms for human-agent cooperation precisely because full autonomy for consequential decisions is not yet trustworthy. Logging and CoT help with debugging and interpretability, but they do not prevent mistakes — only a human checkpoint can catch them before the consequential action fires.

Question 4

Voyager achieves lifelong learning in Minecraft using a skill library. What would be the analogous design pattern for a software engineering agent that needs to accumulate reusable capabilities over time?

  1. Fine-tune the LLM on every new codebase it encounters
  2. Store verified, self-documented code snippets and functions in a retrievable library that the agent can query when facing similar tasks
  3. Give the agent access to GitHub search
  4. Run multiple specialised agents in parallel for different programming languages
Show answer

Correct: Store verified, self-documented code snippets and functions in a retrievable library that the agent can query when facing similar tasks

Voyager's key innovation is exactly this: a skill library of reusable, self-verified programs indexed for retrieval. The direct analogue for a coding agent is a curated, tested snippet library with semantic search — so the agent can retrieve and compose past solutions rather than re-generating from scratch every time.

  1. Without benchmarks, companies cannot comply with AI regulation
  2. Without agreed metrics and test suites, it is impossible to know whether one agent system is genuinely better than another, or just better-demo'd
  3. Evaluation is needed to calculate the cost of running agents at scale

Correct: Without agreed metrics and test suites, it is impossible to know whether one agent system is genuinely better than another, or just better-demo'd

This is the core scientific problem. The survey notes that current evaluations span utility, sociability, values, and continual evolution — but no benchmark covers all four in a standardised way. Without this, the field risks optimising for whatever is easy to demo rather than what actually matters for reliable, safe deployment.
